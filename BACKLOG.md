## Fase 2 — Valor real para review

- 🚧 **13 — Diff Viewer** — unified and side-by-side rendering (`s` toggle), file/hunk navigation, search (`/`, `n`/`N`), whitespace toggle (`w`), open file in `$EDITOR` (`e`), copy the current line (`y`) or the current hunk (`Y`), and a large-diff fallback: any single file's diff over 4,000 lines is truncated for display (stats/hunks still computed from the full diff) with a note pointing at `o` to open the full file in the browser. Side-by-side pairs removal/addition runs block-wise (not full intraline/LCS diffing) and reuses the unified scroll offset as an approximation, so it can drift a little inside hunks with uneven +/- counts; search-match highlighting only applies in unified mode. **Deferred:** context expansion — doing it properly means fetching the full file blob at the diff's SHA and splicing extra unchanged lines into a hunk, a meaningfully bigger feature than the rest of this list; per-file loading state doesn't apply to the current design (the whole diff is fetched in one `glab mr diff` call, not per file).
- ✅ **14 — Pipeline da MR** — stages, jobs, status/duration/ref/source, who started it (`Pipeline.User`, parsed from `glab ci get`'s `user.username`), retry job (`R`), retry pipeline (`P`), cancel pipeline (`x`, confirms), manual job trigger (`t`), and auto-refresh every 10s while the pipeline is running/pending/preparing (`tea.Tick` chain guarded by a generation counter so a stale tick from a screen the user has since left, or a pipeline that's since finished, is a no-op instead of racing a fresh load).
- 🚧 **15 — Logs de Job** — log viewer, search (`/`, `n`/`N`), jump-to-error (`e`/`E`), retry job (`R`), copy the current line (`y`), save the full trace to a file (`s`, writes `job-<id>.log` in the repo root), and follow mode (`f`, toggles on only for a running/pending job; re-fetches both the job's status and trace every 5s and stops itself once the job reaches a terminal status). Very large logs keep only the tail (last 20,000 lines) in memory for display/search — `s` still saves the complete untruncated trace. **Deferred:** paginated download — the `glab api .../trace` endpoint doesn't expose pagination, so "very large" is handled via the display-tail + full-save-on-demand combination above instead.

## Fase 3 — Histórico e produtividade (in progress)

- ✅ **17 — Ações de MR** — approve (`a`), revoke approval (`A`, confirms), draft/ready toggle (`w`), merge (`M`, confirms), checkout branch (`b`, re-detects the local branch via `gitdetect.CurrentBranch` after switching and updates `dash.Branch`). Confirmation required for merge/revoke, matching the plan's "Segurança UX" list. Retry/cancel pipeline and manual job trigger reuse the pipeline-screen actions (Épico 14/15) instead of duplicating them here — there's no job list on the detail screen to pick a specific manual job from. Reviewer/assignee/label management is explicitly "no futuro" per the plan's own Épico 17 task list, not a gap in this scope — deliberately not built.
- ✅ **16 — Discussões e Comentários** — discussions screen (`c` from MR detail). Threads are navigable (`j`/`k`, `g`/`G`); the display policy hides system notes, marks resolvable threads `✓`/`○`, and indents replies. `c` composes a new top-level comment, `R` replies to the currently-selected thread specifically (`POST .../discussions/:id/notes` instead of a new discussion); `alt+enter` inserts a newline while composing so replies aren't forced to one line (plain `enter` still submits). `t` resolves/reopens a thread. Fetching now passes `--paginate` so MRs with more than 100 discussions still load in full. Inline (diff-anchored) comment creation is explicitly "no futuro" per the plan.
- ✅ **18 — Checkout de Branch/MR** — `b` (from MR detail) inspects the working tree first via `gitdetect.HasUncommittedChanges` and, if it's dirty, confirms before `glab mr checkout`; a clean tree checks out directly. The checked-out branch (and its MR) is now recorded into history immediately on success — the dashboard's recent-branches card reflects it right away instead of waiting for the next load.
- 🚧 **19 — Workspace Multi-Repo** — local workspace store (`~/.config/gitlab-tui/workspaces.json`) groups MRs by a branch-derived workspace name (for example `PD-26527`). `W` adds the MR in detail view, `tab` opens the workspace screen, `x` removes the current MR from the selected workspace context, and `Y` copies a consolidated summary. The workspace screen refreshes MR status/pipeline/approvals when possible. **Deferred:** manual workspace naming/editing UI, cross-project MR picker, pending-thread counts, automatic discovery by branch prefix across all accessible projects.
- ✅ **20 — Resumo Copiável** — `Y` copies a paste-ready MR summary in the currently-selected format; `T` cycles the format (plain → Markdown → Slack/Chat-style, shown in the status bar and the hint text). Markdown renders a `## !IID Title` block with a bullet list of facts; Slack/Chat renders the plan's emoji-status single line (`✅`/`❌`/`🟡` by pipeline+approval+conflict state). When the diff or discussions screens have already been visited for the same MR, all three formats also include diff stats (`+adds -dels`) and a discussion count (`N (M resolved)`) — omitted otherwise rather than triggering an extra fetch just for the summary.
- ✅ **21 — Cache Local** — file-backed cache under `~/.cache/gitlab-tui` (or `$XDG_CACHE_HOME/gitlab-tui`) wraps the typed GitLab client when `cache.enabled` is true; it never stores tokens, errors, or job logs. `cache.ttl.*` in config.yaml now sets a retention window per data type (current_user, project, merge_request_list, merge_request_detail, diff, pipeline, pipeline_jobs — duration strings like `"45s"`/`"24h"`, defaulted by `config.Default()` so unset keys keep sensible built-ins); `gzlab cache clear` still invalidates manually. Richer reviewer-recents caching was deliberately skipped — nothing in the app surfaces "recent reviewers" yet (reviewer/assignee management is explicitly future per Épico 17), so caching it now would be speculative.

## Fase 4/5 and polish (backlog)

- 🚧 **22 — UI Base** — responsive shell now has header, footer, and a sidebar/main-content layout on wide terminals, while preserving single-pane rendering on narrower terminals. `ui.theme` applies dark/light palette choices, `ui.mouse` enables Bubble Tea mouse support, and existing confirmation/error/search/loading states are framed by the shared shell. **Deferred:** true modal overlays and command palette.
- **23 — Atalhos** — most of the plan's suggested table is bound now (global: `o`/`y`/`Y`/`T`/`r`/`m`/`f`/`F`/`/`/`q`/`esc`/`j`/`k`/`enter`; diff: `h`/`l`/`[`/`]`/`/`/`n`/`N`/`s`/`w`/`e`/`y`/`Y`; pipeline: `R`/`P`/`t`/`x`; job log: `/`/`n`/`N`/`e`/`E`/`R`/`y`/`s`/`f`; MR detail: `c`/`C`/`a`/`A`/`w`/`M`/`b`/`x`; discussions: `c` (compose)/`R` (reply)/`t` (resolve)/`j`/`k`/`g`/`G`). Deliberately kept `r` = refresh everywhere instead of the plan's per-screen "r = retry job" suggestion, to stay consistent across screens — see BACKLOG notes on 14/15. The `?` help overlay reuses each screen's `*Hints()` (so it never drifts from the footer) and shows the full, untruncated set. `Tab`/`Shift+Tab` are bound (Épico 19: open the workspace screen) rather than the plan's original "cycle focus between panels" — Épico 22 shipped a responsive single-active-screen shell with a sidebar, not multiple simultaneously-focusable panels, so there's no separate panel-focus concept for Tab to cycle through.
- ✅ **24 — CLI Complementar** — `gzlab mr list` (with `--mine`/`--assigned`/`--review` for the cross-project scopes), `mr view [iid]`, `mr checkout <iid>`, `pipeline list` (current branch's MR pipeline + jobs), and `pipeline logs <job-id>` all ship as standalone commands. `pipeline list`/`logs` reuse `internal/tui`'s exported `ParseGLabPipeline`/`ParseGLabJobs` response parsing (so the CLI and the TUI's pipeline screen can't drift apart) behind their own injectable `glabRunner` seam, mirroring the TUI's `Deps.RunGLab`.
- 🚧 **26 — Distribuição** — `make release` builds macOS/Linux/Windows archives with embedded version metadata and checksums; `scripts/install.sh` installs the matching release binary with repo/version/install-dir overrides; README and CHANGELOG document installation and semantic-version release flow. **Deferred:** Homebrew tap and CI-published releases.
